## 一、项目概述

一个轻量级的 A 股数据分析网站，提供三大核心功能：

| 模块 | 功能 | 数据实时性 |
|------|------|-----------|
| 📈 大盘走势 | 三大指数 K 线图（日K/周K/分时），ECharts 可视化 | **实时**（JSONP 直连） |
| 💰 资金流向 | 行业/概念板块主力资金净流入排行 | 15 分钟 |
| 📊 板块分析 | 全量板块涨跌排行（行业 496、概念 495、地域 31），搜索/排序 | 15 分钟 |

## 三、核心技术实现

### 3.1 数据抓取（fetch-data.js）

| 技术点 | 实现方式 |
|--------|---------|
| **数据源** | 东方财富 push2 API（`push2.eastmoney.com/api/qt/clist/get`） |
| **分页抓取** | 东方财富单页上限 100 条，通过 `pn` 参数循环翻页，合并全量数据 |
| **数据类型** | 板块数据（3 类）+ 资金流向（2 类），共 5 个 JSON 文件 |
| **运行环境** | Node.js 20（GitHub Actions 托管） |

**板块数据字段映射**：

| 东方财富字段 | 含义 | 前端用途 |
|-------------|------|---------|
| f12 | 板块代码 | 板块标识 |
| f14 | 板块名称 | 显示名称 |
| f3 | 涨跌幅（需 /100） | 排序、着色 |
| f4 | 涨跌额（需 /100） | 显示 |
| f20 | 总市值 | 板块规模展示 |
| f104/f105 | 上涨/下跌家数 | 市场广度 |
| f128 | 领涨股名称 | 龙头识别 |
| f136 | 领涨股涨幅 | 龙头强度 |

**资金流向字段**：

| 字段 | 含义 |
|------|------|
| f62 | 主力净流入（元） |
| f184 | 主力净流入占比（%） |
| f66 | 超大单净流入 |
| f72 | 大单净流入 |

### 3.2 定时任务（GitHub Actions）

```yaml
cron: '*/15 1-7 * * 1-5'   # UTC 1:00-7:00 = 北京时间 9:00-15:00，周一到周五
```

- **触发逻辑**：cron 定时 + 支持手动触发（`workflow_dispatch`）
- **智能提交**：只在数据有变更时才 commit & push，避免无效部署
- **成本**：免费（GitHub Actions 免费额度 2000 分钟/月，当前每月约 200 分钟）

### 3.3 大盘 K 线（JSONP 实时数据）

| 技术点 | 实现方式 |
|--------|---------|
| **JSONP** | 动态创建 `<script>` 标签，通过 `cb` 参数回调获取数据 |
| **K 线 API** | `push2his.eastmoney.com/api/qt/stock/kline/get` |
| **指数映射** | 上证 `1.000001`、深证 `0.399001`、创业板 `0.399006` |
| **周期支持** | 日K（klt=101, 120天）、周K（klt=102, 52周）、分时（klt=1, 240条） |
| **缓存策略** | 内存 `klineCache` 避免同一指数重复请求 |

**ECharts 图表配置**：
- K 线图：蜡烛图（红涨绿跌）+ 成交量柱状图
- 分时图：折线面积图 + 成交量柱状图
- 交互：dataZoom 滑块缩放、tooltip 十字准星
- 主题：跟随页面明暗模式动态切换

### 3.4 前端交互

| 功能 | 实现方式 |
|------|---------|
| **搜索过滤** | 300ms 防抖，匹配板块名称和代码 |
| **列排序** | 点击表头排序，数值列默认降序，名称列升序，支持中英文排序 |
| **自动刷新** | 交易时间每 15 分钟清除缓存重新 fetch，休市时停止 |
| **明暗主题** | CSS 变量 + `data-theme` 属性切换，localStorage 持久化，首次跟随系统 |
| **响应式** | 768px 断点，移动端隐藏次要列（换手率、领涨股） |

### 3.5 后端（Cloudflare Worker，当前未启用）

| 技术点 | 实现方式 |
|--------|---------|
| **运行时** | Cloudflare Workers（TypeScript） |
| **缓存** | Cloudflare KV，TTL 10 秒 |
| **超时保护** | Promise.race + 5 秒超时，防止东方财富卡死 |
| **诊断接口** | `/ping`（存活检测）、`/health`（KV + API 连通性检查） |
| **数据转换** | 东方财富原始字段码 → 语义化命名（changePercent、leadingStock 等） |
| **当前状态** | ⚠️ 东方财富返回 502，封禁了 Cloudflare 边缘节点 IP |

## 四、开发过程中踩过的坑

### 4.1 GitHub Pages 404

**现象**：前端页面加载数据报 `Unexpected token '<'`

**原因**：GitHub Pages 的 Source 配置不对，应选 "GitHub Actions" 而非 "Deploy from a branch"

**教训**：部署静态站点时，工作流类型和 Pages Source 必须匹配。

### 4.2 fetch-data.js 变量重复声明

**现象**：GitHub Actions 定时任务执行失败

**原因**：`const fs = require('fs')` 和函数内 `const fs = SECTOR_TYPES[type]` 重复声明

**教训**：Node.js 脚本也要做语法检查（`node -c`），不能只测功能。

### 4.3 Cloudflare Worker 访问东方财富 502

**现象**：Worker 请求东方财富 API 返回 502 Bad Gateway

**原因**：东方财富封禁了云服务器/CDN 的 IP 段（Cloudflare 边缘节点），但 GitHub Actions 的 Azure IP 没被封

**教训**：国内金融 API 普遍有反爬策略，不同云厂商 IP 待遇不同。选择数据获取方案时必须实际测试连通性。

### 4.4 JSONP 超时导致 Worker 卡死

**现象**：Worker 所有接口（包括 /health）都无响应

**原因**：`AbortController` 在 Cloudflare Workers 环境下不生效，fetch 永远不返回

**教训**：Workers 环境要用 `Promise.race` + `setTimeout` 实现超时，不能依赖 AbortController。

### 4.5 let 暂时性死区

**现象**：`ReferenceError: Cannot access 'chart' before initialization`

**原因**：`setTheme()` 在脚本顶部执行，但 `let chart` 声明在脚本中部，let 有暂时性死区（TDZ）

**教训**：全局变量必须声明在脚本最顶部，特别是会被初始化代码引用的变量。

### 4.6 东方财富 API 字段陷阱

**现象**：换手率列全部显示 `-`

**原因**：板块级接口不返回 `f8`（换手率），该字段只在个股数据中有效

**教训**：API 文档缺失时，必须先检查实际返回的字段和值，不能想当然。

## 五、可优化项

### 5.1 性能优化

| 优化点 | 当前状态 | 建议方案 |
|--------|---------|---------|
| **JSON 文件体积** | industry.json ~191KB | 只保留前端需要的字段，体积可缩减 60%+ |
| **首次加载** | 所有 JS 在 `<script>` 中一次性加载 | 按 Tab 懒加载 JS 模块，首屏只加载大盘走势代码 |
| **ECharts 体积** | 全量引入 echarts.min.js (~1MB) | 按需引入（只加载 K 线图 + 折线图组件），可缩减到 300KB |
| **JSONP 缓存** | 内存缓存，切换 Tab 丢失 | 用 IndexedDB 持久化 K 线缓存，减少重复请求 |
| **表格渲染** | innerHTML 全量替换 | 板块数据量大时（500 条）可能卡顿，考虑虚拟滚动 |

### 5.2 架构优化

| 优化点 | 当前状态 | 建议方案 |
|--------|---------|---------|
| **单文件 HTML** | 640 行 HTML+CSS+JS 混在一起 | 拆分为独立文件（style.css / app.js / components/），引入 Vite 构建 |
| **数据格式** | 直接存储东方财富原始字段码（f3, f14...） | 抓取时转换为语义化字段，前端代码可读性更好 |
| **错误处理** | JSONP 失败只显示空白 | 增加降级策略：JSONP 失败 → 显示上次缓存数据 + 错误提示 |
| **部署流程** | 每次数据更新触发完整 Pages 部署 | 考虑用 Cloudflare Pages 替代 GitHub Pages，部署更快 |

### 5.3 数据质量优化

| 优化点 | 当前状态 | 建议方案 |
|--------|---------|---------|
| **历史数据** | 无历史数据积累 | 每次抓取同时存一份历史快照，支持板块涨跌历史回看 |
| **数据校验** | 无 | 增加数据完整性检查（字段数量、数值范围），异常数据不入库 |
| **时区处理** | 交易时间判断依赖浏览器时区转换 | 统一用 UTC 计算，避免夏令时等问题 |
| **节假日** | 只判断周末，不判断法定节假日 | 接入中国节假日 API，避免节假日空跑 GitHub Actions |

## 七、技术栈总结

| 层级 | 技术 | 用途 |
|------|------|------|
| **前端** | 原生 HTML/CSS/JS | 单页应用，零依赖 |
